Guía de Estilo BCP API - Especificación OpenAPI v3.0.1

Bienvenido a la Guía de Estilo BCP API. Este documento proporciona una estructura completa y detallada para la creación de especificaciones OpenAPI, asegurando coherencia, calidad y facilidad de mantenimiento en el diseño de nuestras APIs.

1. openapi

Obligatorio: Este campo especifica la versión de la especificación OpenAPI que se está utilizando. Es crucial para asegurar la compatibilidad y correcta interpretación del documento por herramientas y bibliotecas.

Ejemplo:

openapi: 3.0.1

2. Info

Obligatorio: Proporciona metadatos sobre la API, incluyendo título, descripción, términos de servicio, información de contacto, licencia y versión. Estos datos son esenciales para que los consumidores de la API comprendan su propósito y cómo utilizarla adecuadamente.

Subsecciones:

  • 2.1. Title
  • 2.2. Description
  • 2.3. Terms of Service
  • 2.4. Contact
  • 2.5. License
  • 2.6. Version

2.1. Title

Obligatorio: Nombre claro y conciso de la API. Debe reflejar su propósito principal.

Ejemplo:

title: BCP API Guide

2.2. Description

Obligatorio: Descripción detallada de la API, sus funcionalidades principales y casos de uso. Utiliza formato de texto enriquecido (multilínea) para mayor claridad.

Ejemplo:

description: |
  Esta API permite gestionar los empleados de BCP, incluyendo operaciones para crear, leer, actualizar y eliminar registros de empleados.

3. Servers

Opcional: Define las URLs de los servidores donde está desplegada la API. Permite especificar diferentes entornos como producción y sandbox.

Ejemplo:

servers:
  - url: https://api.bcp.com.pe/v1
    description: Servidor de Producción
  - url: https://sandbox.api.bcp.com.pe/v1
    description: Servidor de Sandbox

4. Paths

Obligatorio: Define los endpoints disponibles en la API y los métodos HTTP que soportan (GET, POST, PUT, DELETE).

Ejemplo:

paths:
  /employees:
    get:
      summary: Obtener lista de empleados
      responses:
        '200':
          description: Lista obtenida exitosamente

5. Components

Opcional: Contiene definiciones reutilizables que pueden ser referenciadas en diferentes partes de la especificación.

Ejemplo de Esquema:

schemas:
  Employee:
    type: object
    required:
      - id
      - name
    properties:
      id:
        type: integer
      name:
        type: string

6. Security

Opcional: Define los mecanismos de autenticación y autorización que se aplican globalmente a la API.

Ejemplo:

security:
  - ApiKeyAuth: []
  - OAuth2:
      - read

9. Ejemplo Completo

Este es un ejemplo completo de una especificación OpenAPI basada en un CRUD de empleados.

openapi: 3.0.1
info:
  title: BCP Employee Management API
  version: 1.0.0
paths:
  /employees:
    get:
      summary: Lista de empleados
      responses:
        '200':
          description: Operación exitosa
